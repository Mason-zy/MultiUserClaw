• 结论

  - bridge 本身不会因为前端断开 SSE/页面切换就自动中止任务。它只有两个明确的 abort 入口，会主动发 chat.abort：
    openclaw/bridge/routes/sessions.ts:196 和 openclaw/bridge/routes/sessions.ts:218。
    SSE 断开时只做 offEvent(listener)，没有 abort：
    openclaw/bridge/routes/events.ts:161
  - 当前机器上的实际 OpenClaw 配置把超时设得很短：
    /Users/admin/.openclaw/openclaw.json:47
    里 subagents.runTimeoutSeconds = 600，timeoutSeconds = 300。
    也就是主 Agent 5 分钟，子 Agent 10 分钟。
  - Gateway 在 chat.send 时会把这个 timeout 转成 run 的 expiresAtMs：
    openclaw/src/gateway/server-methods/chat.ts:1501
    和 openclaw/src/gateway/server-methods/chat.ts:1602。
    后台维护线程到点后会直接 abortChatRunById(..., stopReason: "timeout")：
    openclaw/src/gateway/server-maintenance.ts:108
  - 这就是你们看到“长任务中断 + 模型日志很多 Abort”的最像根因。不是 bridge 在乱发 abort，而是 run 超时后，abort signal 会一路传到 reply
    pipeline、主 Agent、子 Agent、模型流，所以一次 timeout 会扇出很多 Abort 日志：
    openclaw/src/auto-reply/reply/reply-run-registry.ts:245
    和 openclaw/src/agents/pi-embedded-runner/run/attempt.ts:1542
  - 还有一个前端侧“看起来像中断”的问题：聊天页本地等待也有硬上限。
    frontend 10 分钟：
    frontend/src/pages/Chat.tsx:578
    simple_front 4 分钟：
    simple_front/src/pages/Chat.tsx:437
    share_openclaw_front 2 分钟：
    share_openclaw_front/src/pages/Chat.tsx:227
    这个不会真的 abort 后端，但会让用户先感觉“任务停了”。

  判断

  如果线上容器的 ~/.openclaw/openclaw.json 也是这组值，那么主因基本就是超时配置，不是 bridge 代码。bridge 还会保留已有 timeout，不会帮你改掉它：
  openclaw/bridge/config.ts:141
