# 老服务器无感迁移方案与操作步骤

本文用于把老服务器上的 MultiUserClaw / Hermes 用户完整迁移到新服务器，并尽量做到用户侧无感：

- 用户账号、`user_id`、登录身份不变。
- Hermes 历史会话、主聊天窗口、session key、workspace 文件不变。
- 飞书机器人、websocket 长连接、定时任务在新服务器继续工作。
- 新旧服务器不同时运行同一个用户的 Hermes 容器，避免飞书重复回复和 cron 重复执行。

本文默认新服务器已经可以从私有仓库拉取 `main` 并完成基础部署。空部署教程见：

```text
doc/tec/新机器私有仓库部署教程.md
```

## 1. 迁移对象

MultiUserClaw 的用户状态不只在数据库里。要无感迁移，至少要迁移这些对象：

```text
Postgres:
  users
  user_port_bindings
  usage_records
  audit_logs
  以及与目标用户直接相关的运行记录

Docker volume:
  hermes-data-<uid8>        -> 容器内 /workspace
  hermes-data-<uid8>-home   -> 容器内 /opt/data

Hermes 关键文件:
  /opt/data/config.yaml
  /opt/data/.env
  /opt/data/sessions/
  /opt/data/profiles/
  /opt/data/cron/
  /opt/data/logs/
  /opt/data/auth.json
  /workspace/*
```

用户容器本身不建议迁移。正确做法是迁移数据库记录和 volume，让新服务器 gateway 基于本地镜像重新创建 `hermes-user-<uid8>` 容器。

## 2. 核心原则

### 2.1 保留用户 ID

无感迁移必须保留 `users.id`。Hermes 容器名、volume 名、端口绑定、会话代理都依赖这个 ID 的前 8 位：

```text
user id:    9c0d224f-d029-4d61-890d-ac74a2d69e72
uid8:       9c0d224f
container:  hermes-user-9c0d224f
volumes:    hermes-data-9c0d224f
            hermes-data-9c0d224f-home
```

如果重新注册出新 `user_id`，历史主窗口和飞书 session 会分裂，不算无感迁移。

### 2.2 旧机先停，新机再起

飞书机器人使用 websocket 模式时，同一个 App ID / App Secret 不能让旧机和新机同时稳定处理消息。迁移切换点必须执行：

```bash
sudo docker stop hermes-user-<uid8>
```

确认旧容器停掉后，再在新机启动对应用户容器。

### 2.3 不导入旧 containers 记录

`containers` 表保存的是旧服务器上的 Docker container id、内网 IP、状态等运行态信息。迁移到新机器时，这些值必然失效。

推荐做法：

- 迁移 `users`、`user_port_bindings`、历史审计和 usage 数据。
- 不迁移旧 `containers` 表记录，或导入前删除目标用户的旧 `containers` 记录。
- 让新服务器 gateway 重新创建容器并写入新的 container id / internal host。

### 2.4 密钥和模型配置要跟随新机平台

用户 Hermes 容器内的飞书配置要从老机保留：

```text
/opt/data/.env 中的 FEISHU_*
/opt/data/auth.json
/opt/data/sessions/*
```

平台级 LLM 配置建议使用新机当前标准：

```env
PLATFORM_DEDICATED_HERMES_DEFAULT_PROVIDER=platform-gateway
DEFAULT_MODEL=openai/glm-5.1
OPENAI_API_BASE=https://fjbigmodel.fjdac.cn/v1
```

迁移后确认用户容器 `/opt/data/config.yaml` 中：

```yaml
model:
  provider: platform-gateway
  default: openai/glm-5.1
```

## 3. 迁移前检查

以下命令中的变量按实际环境替换：

```bash
OLD_HOST=<旧服务器IP或域名>
OLD_USER=<旧服务器SSH用户>
TARGET_EMAIL=<要迁移的用户邮箱>
```

### 3.1 老服务器确认用户

在旧服务器执行：

```bash
cd /path/to/MultiUserClaw

sudo docker exec openclaw-postgres psql -U nanobot -d nanobot_platform -x -c "
SELECT
  u.id,
  u.username,
  u.email,
  u.is_active,
  c.name AS container_name,
  c.status,
  c.internal_host,
  c.internal_port,
  c.last_active_at
FROM users u
LEFT JOIN containers c ON c.user_id = u.id
WHERE u.email = '${TARGET_EMAIL}'
   OR u.username = '${TARGET_EMAIL}';
"
```

记录：

```text
USER_ID=<完整 UUID>
UID8=<USER_ID 前 8 位>
CONTAINER=hermes-user-<UID8>
```

### 3.2 老服务器确认容器和 volume

```bash
sudo docker ps -a --format 'table {{.Names}}\t{{.Status}}\t{{.Image}}\t{{.Ports}}' | grep -E "NAMES|hermes-user-${UID8}"

sudo docker volume ls --format '{{.Name}}' | grep -E "^hermes-data-${UID8}($|-home$)"
```

必须看到：

```text
hermes-data-<UID8>
hermes-data-<UID8>-home
```

### 3.3 老服务器确认飞书和定时任务

```bash
sudo docker exec "hermes-user-${UID8}" sh -lc '
set -e
echo "== config =="
grep -n "provider\\|default" /opt/data/config.yaml || true
echo "== feishu env =="
grep -E "^FEISHU_|^LARK_" /opt/data/.env || true
echo "== sessions =="
find /opt/data/sessions -maxdepth 2 -type f | sort | head -50
echo "== cron =="
find /opt/data -path "*/cron/*" -type f | sort | head -50
'
```

业务日志不要只看 `docker logs`，要看容器内文件日志：

```bash
sudo docker exec "hermes-user-${UID8}" tail -n 200 /opt/data/logs/gateway.log
sudo docker exec "hermes-user-${UID8}" tail -n 100 /opt/data/logs/errors.log
```

## 4. 新服务器基础准备

新服务器先按 `doc/tec/新机器私有仓库部署教程.md` 完成部署，再检查：

```bash
cd /home/ubuntu/Project/MultiUserClaw

git status --short --branch
git log -1 --oneline --decorate

sudo docker compose ps
sudo docker images | grep -E 'hermes-base|nanobot-hermes-agent'

curl -sS http://127.0.0.1:9900/api/ping
```

检查 gateway 环境变量：

```bash
sudo docker exec openclaw-gateway env | grep -E \
'PLATFORM_DEFAULT_MODEL|PLATFORM_DEDICATED_HERMES_DEFAULT_PROVIDER|PLATFORM_OPENAI_API_BASE'
```

预期至少包含：

```text
PLATFORM_DEFAULT_MODEL=openai/glm-5.1
PLATFORM_DEDICATED_HERMES_DEFAULT_PROVIDER=platform-gateway
PLATFORM_OPENAI_API_BASE=https://fjbigmodel.fjdac.cn/v1
```

如果新服务器已有同名用户或同 `user_id` 的半迁移数据，先备份再清理，避免和旧数据混合。

## 5. 导出老服务器数据

以下命令建议在旧服务器项目目录下执行，导出包放到临时目录。

```bash
export USER_ID=<完整 UUID>
export UID8=<USER_ID 前 8 位>
export MIG_DIR="/tmp/muc-migrate-${UID8}"

mkdir -p "${MIG_DIR}"
```

### 5.1 导出用户数据库记录

导出前先确认表结构。不同版本可能有少量表差异，先列出相关表：

```bash
sudo docker exec openclaw-postgres psql -U nanobot -d nanobot_platform -c "\dt"
```

推荐使用 `\copy` 导出目标用户相关数据：

```bash
sudo docker exec -i openclaw-postgres psql -U nanobot -d nanobot_platform -v user_id="${USER_ID}" \
  > "${MIG_DIR}/users.csv" <<'SQL'
\copy (SELECT * FROM users WHERE id = :'user_id') TO STDOUT WITH CSV HEADER
SQL

sudo docker exec -i openclaw-postgres psql -U nanobot -d nanobot_platform -v user_id="${USER_ID}" \
  > "${MIG_DIR}/user_port_bindings.csv" <<'SQL'
\copy (SELECT * FROM user_port_bindings WHERE user_id = :'user_id') TO STDOUT WITH CSV HEADER
SQL

sudo docker exec -i openclaw-postgres psql -U nanobot -d nanobot_platform -v user_id="${USER_ID}" \
  > "${MIG_DIR}/usage_records.csv" <<'SQL'
\copy (SELECT * FROM usage_records WHERE user_id = :'user_id') TO STDOUT WITH CSV HEADER
SQL

sudo docker exec -i openclaw-postgres psql -U nanobot -d nanobot_platform -v user_id="${USER_ID}" \
  > "${MIG_DIR}/audit_logs.csv" <<'SQL'
\copy (SELECT * FROM audit_logs WHERE user_id = :'user_id') TO STDOUT WITH CSV HEADER
SQL
```

如果某张表不存在，说明当前版本没有这类数据，记录下来即可，不要中断整个迁移。

不要导出旧 `containers` 记录作为新机最终状态。最多只导出作审计备查：

```bash
sudo docker exec -i openclaw-postgres psql -U nanobot -d nanobot_platform -v user_id="${USER_ID}" \
  > "${MIG_DIR}/containers.old.csv" <<'SQL'
\copy (SELECT * FROM containers WHERE user_id = :'user_id') TO STDOUT WITH CSV HEADER
SQL
```

### 5.2 停止旧用户容器

这是正式切换点。停掉旧容器后，飞书 websocket 和 cron 不再由旧服务器执行。

```bash
sudo docker stop "hermes-user-${UID8}"
```

确认停止：

```bash
sudo docker ps -a --format 'table {{.Names}}\t{{.Status}}' | grep -E "NAMES|hermes-user-${UID8}"
```

### 5.3 导出用户 Docker volumes

```bash
for v in "hermes-data-${UID8}" "hermes-data-${UID8}-home"; do
  sudo docker run --rm \
    -v "${v}:/data:ro" \
    -v "${MIG_DIR}:/backup" \
    alpine sh -lc "cd /data && tar czf /backup/${v}.tgz ."
done
```

生成校验文件：

```bash
cd "${MIG_DIR}"
sha256sum *.csv *.tgz > SHA256SUMS
ls -lh
```

## 6. 传输到新服务器

从新服务器拉取或从旧服务器推送都可以。示例为从新服务器拉取：

```bash
export OLD_HOST=<旧服务器IP或域名>
export OLD_USER=<旧服务器SSH用户>
export UID8=<USER_ID 前 8 位>

mkdir -p "/tmp/muc-migrate-${UID8}"
rsync -avz "${OLD_USER}@${OLD_HOST}:/tmp/muc-migrate-${UID8}/" "/tmp/muc-migrate-${UID8}/"
```

在新服务器校验：

```bash
cd "/tmp/muc-migrate-${UID8}"
sha256sum -c SHA256SUMS
```

## 7. 新服务器恢复数据

以下命令在新服务器执行。

```bash
cd /home/ubuntu/Project/MultiUserClaw

export USER_ID=<完整 UUID>
export UID8=<USER_ID 前 8 位>
export MIG_DIR="/tmp/muc-migrate-${UID8}"
```

### 7.1 停止并移除新服务器同名旧容器

如果新服务器上已经有同 `uid8` 容器，先删除容器，但不要误删刚迁移来的包。

```bash
sudo docker rm -f "hermes-user-${UID8}" 2>/dev/null || true
```

如果新服务器已经有同名 volume，先备份再删除，避免混合数据：

```bash
for v in "hermes-data-${UID8}" "hermes-data-${UID8}-home"; do
  if sudo docker volume inspect "$v" >/dev/null 2>&1; then
    sudo docker run --rm \
      -v "${v}:/data:ro" \
      -v "${MIG_DIR}:/backup" \
      alpine sh -lc "cd /data && tar czf /backup/${v}.pre-migrate-new-server.tgz ."
    sudo docker volume rm "$v"
  fi
done
```

### 7.2 恢复 Docker volumes

```bash
for v in "hermes-data-${UID8}" "hermes-data-${UID8}-home"; do
  sudo docker volume create "$v"
  sudo docker run --rm \
    -v "${v}:/data" \
    -v "${MIG_DIR}:/backup:ro" \
    alpine sh -lc "cd /data && tar xzf /backup/${v}.tgz"
done
```

确认文件已恢复：

```bash
sudo docker run --rm -v "hermes-data-${UID8}-home:/data:ro" alpine \
  sh -lc 'ls -la /data | head && find /data/sessions -maxdepth 2 -type f 2>/dev/null | head'
```

### 7.3 导入数据库记录

先备份新服务器当前数据库：

```bash
sudo docker exec openclaw-postgres pg_dump -U nanobot -d nanobot_platform \
  > "${MIG_DIR}/new-server-before-user-${UID8}.sql"
```

把 CSV 放入 Postgres 容器：

```bash
sudo docker cp "${MIG_DIR}/users.csv" openclaw-postgres:/tmp/users.csv
sudo docker cp "${MIG_DIR}/user_port_bindings.csv" openclaw-postgres:/tmp/user_port_bindings.csv
sudo docker cp "${MIG_DIR}/usage_records.csv" openclaw-postgres:/tmp/usage_records.csv
sudo docker cp "${MIG_DIR}/audit_logs.csv" openclaw-postgres:/tmp/audit_logs.csv
```

导入前删除目标用户在新服务器上的旧记录。删除顺序按依赖从子表到主表：

```bash
sudo docker exec openclaw-postgres psql -U nanobot -d nanobot_platform -v user_id="${USER_ID}" -c "
BEGIN;
DELETE FROM containers WHERE user_id = :'user_id';
DELETE FROM runtime_runs WHERE user_id = :'user_id';
DELETE FROM usage_records WHERE user_id = :'user_id';
DELETE FROM audit_logs WHERE user_id = :'user_id';
DELETE FROM user_port_bindings WHERE user_id = :'user_id';
DELETE FROM users WHERE id = :'user_id';
COMMIT;
"
```

如果某张表不存在，去掉对应 `DELETE` 后重跑。

导入 CSV：

```bash
sudo docker exec -i openclaw-postgres psql -U nanobot -d nanobot_platform <<'SQL'
BEGIN;
\copy users FROM '/tmp/users.csv' WITH CSV HEADER;
\copy user_port_bindings FROM '/tmp/user_port_bindings.csv' WITH CSV HEADER;
\copy usage_records FROM '/tmp/usage_records.csv' WITH CSV HEADER;
\copy audit_logs FROM '/tmp/audit_logs.csv' WITH CSV HEADER;
COMMIT;
SQL
```

核验导入结果：

```bash
sudo docker exec openclaw-postgres psql -U nanobot -d nanobot_platform -x -v user_id="${USER_ID}" -c "
SELECT id, username, email, is_active, created_at, updated_at
FROM users
WHERE id = :'user_id';
"

sudo docker exec openclaw-postgres psql -U nanobot -d nanobot_platform -v user_id="${USER_ID}" -c "
SELECT 'user_port_bindings' AS table_name, count(*) FROM user_port_bindings WHERE user_id = :'user_id'
UNION ALL
SELECT 'usage_records', count(*) FROM usage_records WHERE user_id = :'user_id'
UNION ALL
SELECT 'audit_logs', count(*) FROM audit_logs WHERE user_id = :'user_id'
UNION ALL
SELECT 'containers', count(*) FROM containers WHERE user_id = :'user_id';
"
```

`containers` 这里预期是 `0`，等待新服务器重新创建。

## 8. 新服务器启动用户容器

触发用户容器创建有两种方式。

### 8.1 用户正常登录触发

让用户访问新服务器前端并登录。登录成功后访问聊天页，gateway 会按需创建 `hermes-user-<uid8>`。

### 8.2 管理员用临时 token 触发

在 gateway 容器里生成该用户 JWT：

```bash
TOKEN=$(
  sudo docker exec -i openclaw-gateway python - <<PY
from app.auth.service import create_access_token
print(create_access_token("${USER_ID}", "user"))
PY
)
```

调用用户侧接口，触发 gateway 确保容器存在：

```bash
curl -sS -H "Authorization: Bearer ${TOKEN}" \
  http://127.0.0.1:9900/api/openclaw/sessions | head
```

检查容器：

```bash
sudo docker ps -a --format 'table {{.Names}}\t{{.Status}}\t{{.Image}}\t{{.Ports}}' | grep -E "NAMES|hermes-user-${UID8}"
```

## 9. 迁移后验证

### 9.1 数据库和容器

```bash
sudo docker exec openclaw-postgres psql -U nanobot -d nanobot_platform -x -v user_id="${USER_ID}" -c "
SELECT
  u.id,
  u.username,
  u.email,
  u.is_active,
  c.status,
  c.name,
  c.internal_host,
  c.internal_port,
  c.last_active_at
FROM users u
LEFT JOIN containers c ON c.user_id = u.id
WHERE u.id = :'user_id';
"
```

### 9.2 Hermes 健康检查

```bash
sudo docker exec "hermes-user-${UID8}" curl -sS http://127.0.0.1:18080/health
```

### 9.3 模型配置

```bash
sudo docker exec "hermes-user-${UID8}" sh -lc '
grep -n "provider\\|default\\|base_url" /opt/data/config.yaml
'
```

预期：

```text
provider: platform-gateway
default: openai/glm-5.1
```

### 9.4 历史主窗口和会话

```bash
TOKEN=$(
  sudo docker exec -i openclaw-gateway python - <<PY
from app.auth.service import create_access_token
print(create_access_token("${USER_ID}", "user"))
PY
)

curl -sS -H "Authorization: Bearer ${TOKEN}" \
  http://127.0.0.1:9900/api/openclaw/sessions
```

确认返回的会话标题、session id、飞书 session key 与旧服务器一致。

容器内也可以直接看：

```bash
sudo docker exec "hermes-user-${UID8}" find /opt/data/sessions -maxdepth 2 -type f | sort | head -100
```

### 9.5 飞书 websocket

先看 stdout 启动日志：

```bash
sudo docker logs --tail 200 "hermes-user-${UID8}" | grep -Ei 'feishu|lark|websocket|connected'
```

预期能看到类似：

```text
[Feishu] Connected in websocket mode
feishu connected
```

再让用户在飞书给机器人发一条测试消息，立刻看业务日志：

```bash
sudo docker exec "hermes-user-${UID8}" tail -n 200 /opt/data/logs/gateway.log
sudo docker exec "hermes-user-${UID8}" tail -n 200 /opt/data/logs/agent.log
sudo docker exec "hermes-user-${UID8}" tail -n 100 /opt/data/logs/errors.log
```

判断链路：

```text
gateway.log 有 inbound message / response ready / Sending response
agent.log 有 provider / API call / latency
errors.log 没有新的异常堆栈
```

### 9.6 定时任务

```bash
sudo docker exec -u hermes "hermes-user-${UID8}" \
  /opt/hermes/.venv/bin/hermes cron list
```

如果 CLI 输出 `Gateway is not running`，不要直接判定失败。以以下两项为准：

```bash
sudo docker exec "hermes-user-${UID8}" ps aux | grep -E 'gateway|hermes' | grep -v grep
sudo docker exec "hermes-user-${UID8}" grep -Ei 'cron scheduler|next run|started' /opt/data/logs/gateway.log /opt/data/logs/agent.log
```

## 10. 入口切换

如果用户通过 IP 访问：

```text
用户前端: http://<新服务器公网IP>:8180
管理后台: http://<新服务器公网IP>:8181
gateway:  http://<新服务器公网IP>:9900
```

如果有域名或 Nginx：

1. 新服务器先本地验证 `8180`、`8181`、`9900`。
2. Nginx upstream 改到新服务器。
3. DNS 或负载均衡切换到新服务器。
4. 切换后观察 `openclaw-gateway` 和目标 `hermes-user-*` 日志。

平台登录 session 是否继续有效取决于浏览器 token、域名和新旧服务器的 `JWT_SECRET`。如果 `JWT_SECRET` 不同，用户可能需要重新登录；这不影响 Hermes 历史会话、飞书会话和主窗口数据。

## 11. 回滚方案

迁移完成前不要删除旧服务器 volume。推荐至少保留一个回滚窗口。

### 11.1 回滚到旧服务器

1. 停止新服务器目标用户容器：

```bash
sudo docker stop "hermes-user-${UID8}"
```

2. 在旧服务器启动原容器：

```bash
sudo docker start "hermes-user-${UID8}"
```

3. 如果已切域名或反代，把入口切回旧服务器。

4. 让用户在飞书发测试消息，确认旧服务器日志有新消息。

### 11.2 新服务器数据回滚

如果新服务器导入错误，可以用导入前备份恢复：

```bash
sudo docker exec -i openclaw-postgres psql -U nanobot -d nanobot_platform \
  < "/tmp/muc-migrate-${UID8}/new-server-before-user-${UID8}.sql"
```

也可以只删除目标用户新导入的数据：

```bash
sudo docker rm -f "hermes-user-${UID8}" 2>/dev/null || true
sudo docker volume rm "hermes-data-${UID8}" "hermes-data-${UID8}-home"

sudo docker exec openclaw-postgres psql -U nanobot -d nanobot_platform -v user_id="${USER_ID}" -c "
BEGIN;
DELETE FROM containers WHERE user_id = :'user_id';
DELETE FROM runtime_runs WHERE user_id = :'user_id';
DELETE FROM usage_records WHERE user_id = :'user_id';
DELETE FROM audit_logs WHERE user_id = :'user_id';
DELETE FROM user_port_bindings WHERE user_id = :'user_id';
DELETE FROM users WHERE id = :'user_id';
COMMIT;
"
```

## 12. 常见问题

### 12.1 迁移后用户登录密码变了

如果导入的是旧服务器 `users.password_hash`，新服务器上的平台登录密码就是旧服务器密码。

如果需要重置，不要重新建用户。直接保持 `users.id` 不变，只更新该用户密码 hash。

### 12.2 飞书机器人不回复

按这个顺序排查：

1. 确认旧服务器同用户容器已经停止。
2. 看新服务器 `docker logs hermes-user-<uid8>` 是否显示 websocket connected。
3. 看容器内 `/opt/data/logs/gateway.log`，不要只看 `docker logs`。
4. 看 `/opt/data/.env` 是否保留 `FEISHU_*`。
5. 看 `/opt/data/logs/agent.log` 是否有 LLM provider 调用和错误。

### 12.3 LLM 401

优先检查：

```bash
sudo docker exec "hermes-user-${UID8}" cat /opt/data/config.yaml
sudo docker exec openclaw-gateway env | grep PLATFORM_DEDICATED_HERMES_DEFAULT_PROVIDER
```

正确 provider 是：

```text
platform-gateway
```

如果用户容器里还是 `auto` 或其他 provider，删除用户容器让 gateway 重建。不要删除用户 volume：

```bash
sudo docker rm -f "hermes-user-${UID8}"
```

### 12.4 会话列表为空

检查 volume 是否恢复到正确位置：

```bash
sudo docker exec "hermes-user-${UID8}" sh -lc '
ls -la /opt/data
find /opt/data/sessions -maxdepth 2 -type f | sort | head -100
'
```

如果 `/opt/data/sessions` 缺失，通常是 `hermes-data-<uid8>-home` 没恢复或恢复到了错误 volume。

### 12.5 管理后台看到容器旧状态

不要迁移旧 `containers` 记录。删除目标用户 `containers` 记录后，让 gateway 在新服务器重新创建：

```bash
sudo docker exec openclaw-postgres psql -U nanobot -d nanobot_platform -v user_id="${USER_ID}" -c "
DELETE FROM containers WHERE user_id = :'user_id';
"
```

再触发用户接口或让用户登录。

## 13. 一次迁移完成后的核对清单

- 新服务器代码是私有仓库 `main` 的目标提交。
- 新服务器 `openclaw-gateway`、`openclaw-frontend`、`openclaw-manage`、`openclaw-postgres` 正常。
- 新服务器存在 `hermes-base:latest` 和 `nanobot-hermes-agent:latest`。
- `PLATFORM_DEDICATED_HERMES_DEFAULT_PROVIDER=platform-gateway` 已注入 gateway。
- 目标用户 `users.id` 与旧服务器一致。
- 目标用户两个 volume 都已恢复。
- 旧服务器同用户容器已停止。
- 新服务器 `hermes-user-<uid8>` 正常运行。
- `/health` 返回正常。
- `/opt/data/config.yaml` provider 是 `platform-gateway`。
- `/opt/data/.env` 保留飞书配置。
- `/opt/data/sessions/` 有历史会话。
- 前端 `/api/openclaw/sessions` 能返回历史主窗口。
- 飞书 websocket connected。
- 飞书测试消息有 inbound、LLM 调用、Sending response 日志。
- cron list 或日志能看到定时任务已加载。
- 旧服务器 volume 和导出包暂时保留，回滚窗口结束后再清理。
