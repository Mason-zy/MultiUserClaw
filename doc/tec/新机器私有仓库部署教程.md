# 新机器从私有仓库部署教程

本文用于在一台全新的服务器上，从当前私有仓库拉取 `main` 分支并部署 MultiUserClaw / Hermes 多用户平台。

适用场景：

- 新机器首次部署，数据库和用户容器数据都是空的。
- 从旧机器迁移到新机器，但代码从私有仓库重新拉取。
- 需要按当前 Hermes runtime 方案部署，而不是旧 OpenClaw runtime。

## 1. 关键结论

不要只执行 `docker compose up -d`。

本项目除了 compose 中固定的 `postgres`、`gateway`、`frontend`、`manage-front`、`simple-front` 服务外，还会由 `gateway` 通过 Docker socket 动态创建 `hermes-user-*` 用户容器。因此新机器必须先构建好本地用户 runtime 镜像：

```bash
hermes-base:latest
nanobot-hermes-agent:latest
```

推荐使用项目自带脚本：

```bash
bash build_base_image.sh
python3 deploy_docker.py --host <公网IP或域名> --gateway-port 9900 --frontend-port 8180
```

## 2. 新机器前置条件

以 Ubuntu 为例：

```bash
sudo apt update
sudo apt install -y git python3 docker.io docker-compose-plugin docker-buildx
sudo systemctl enable --now docker
```

确认 Docker 可用：

```bash
docker --version
docker compose version
docker info
```

如果当前用户不是 root，建议加入 docker 组后重新登录：

```bash
sudo usermod -aG docker "$USER"
```

## 3. 拉取私有仓库代码

先确保新机器已经配置 GitHub 私有仓库访问权限，例如 SSH key 或可用的 HTTPS token。

推荐 SSH 方式：

```bash
git clone git@github.com:Mason-zy/MultiUserClaw-private.git
cd MultiUserClaw
git checkout main
git pull origin main
```

如果使用 HTTPS，需要确保凭证可访问私有仓库：

```bash
git clone https://github.com/Mason-zy/MultiUserClaw-private.git
cd MultiUserClaw
git checkout main
git pull origin main
```

检查当前分支和提交：

```bash
git status --short --branch
git log -1 --oneline --decorate
```

## 4. 配置 `.env`

`.env` 不会随 Git 提交，新机器必须重新创建：

```bash
cp .env.example .env
```

编辑 `.env`，至少配置管理员账号、JWT secret 和一个 LLM provider。

当前公司 fj bigmodel 方案建议配置：

```env
ADMIN_USERNAME=admin
ADMIN_PASSWORD=<改成强密码>
JWT_SECRET=<随机强密钥>

OPENAI_API_KEY=<公司 fj bigmodel key>
OPENAI_API_BASE=https://fjbigmodel.fjdac.cn/v1
DEFAULT_MODEL=openai/glm-5.1

PLATFORM_DEDICATED_RUNTIME_BACKEND=hermes
PLATFORM_SHARED_RUNTIME_BACKEND=hermes
PLATFORM_HERMES_API_TOOLSETS=terminal,file,skills
PLATFORM_HERMES_REASONING_EFFORT=none
```

生成 JWT secret 的命令：

```bash
python3 -c "import secrets; print(secrets.token_urlsafe(32))"
```

## 5. 确认 `docker-compose.override.yml`

当前部署依赖 `docker-compose.override.yml` 中的 gateway 环境变量：

```yaml
PLATFORM_DEDICATED_HERMES_DEFAULT_PROVIDER: platform-gateway
```

这条配置很重要。它会让新建的 `hermes-user-*` 容器生成：

```yaml
model:
  provider: platform-gateway
  default: openai/glm-5.1
```

如果缺少这条，Hermes 可能回退到错误 provider，导致 LLM 401。

当前 override 还追加了端口：

```text
8180 -> 用户前端
8181 -> 管理后台
8182 -> simple-front
```

注意：Docker Compose 的 `ports` 是追加，不是替换。原始 `3080`、`3081`、`3082` 也仍会映射。

## 6. 构建和启动

首次部署先构建 Hermes base：

```bash
bash build_base_image.sh
```

再启动整套服务：

```bash
python3 deploy_docker.py --host <公网IP或域名> --gateway-port 9900 --frontend-port 8180
```

如果前面有 Nginx / HTTPS 反向代理，建议前端使用相对 API 路径：

```bash
python3 deploy_docker.py --host <域名> --gateway-port 9900 --frontend-port 8180 --relative-api
```

如果机器网络访问 PyPI、Docker Hub、GitHub 较慢，可以先设置镜像源相关环境变量：

```bash
export APT_DEBIAN_MIRROR=http://mirrors.ustc.edu.cn/debian
export APT_SECURITY_MIRROR=http://mirrors.ustc.edu.cn/debian-security
export PIP_INDEX_URL=https://pypi.tuna.tsinghua.edu.cn/simple
```

需要浏览器工具时，再显式启用 Playwright Chromium：

```bash
PLAYWRIGHT_DOWNLOAD_HOST=https://npmmirror.com/mirrors/playwright \
python3 deploy_docker.py --rebuild hermes --with-browser
```

## 7. 端口和安全组

当前默认端口：

```text
8180 / 3080  用户前端
8181 / 3081  管理后台
8182 / 3082  simple-front
9900         gateway
9901         postgres
```

建议安全组：

```text
8180  按需开放
8181  只给管理员 IP 开放
8182  按需开放
9900  只给前端、反代或调试 IP 开放
9901  不要公网开放
```

如果使用 Nginx，推荐只暴露 80/443，由 Nginx 转发到本机 `8180`、`8181`、`9900`。

## 8. 启动后验证

检查 compose 服务：

```bash
docker compose ps
```

检查 gateway：

```bash
curl http://127.0.0.1:9900/api/ping
```

检查关键镜像：

```bash
docker images | grep -E 'hermes-base|nanobot-hermes-agent'
```

检查 gateway 环境变量：

```bash
docker exec openclaw-gateway env | grep -E 'PLATFORM_DEFAULT_MODEL|PLATFORM_DEDICATED_HERMES_DEFAULT_PROVIDER|PLATFORM_OPENAI_API_BASE'
```

预期包含：

```text
PLATFORM_DEFAULT_MODEL=openai/glm-5.1
PLATFORM_DEDICATED_HERMES_DEFAULT_PROVIDER=platform-gateway
PLATFORM_OPENAI_API_BASE=https://fjbigmodel.fjdac.cn/v1
```

注册或创建一个新用户后，确认用户容器：

```bash
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Image}}' | grep hermes-user
```

检查用户容器模型配置：

```bash
docker exec <hermes-user-容器名> sh -lc 'grep -n "provider\\|default" /opt/data/config.yaml'
```

预期：

```yaml
provider: platform-gateway
default: openai/glm-5.1
```

## 9. 数据初始化机制

`gateway` 首次启动会自动做这些事：

- 创建数据库表。
- 补齐 ORM 中新增的简单字段。
- 创建 `.env` 中的管理员账号。
- 当 `model_provider_configs` 表为空时，从环境变量 seed 模型配置。

注意：模型配置 seed 只在 `model_provider_configs` 表为空时执行。也就是说：

- 全新数据库：`.env` 中的模型配置会初始化进 DB。
- 迁移旧数据库：旧 DB 的模型配置优先，修改 `.env DEFAULT_MODEL` 不会覆盖已有 DB 配置。

## 10. 迁移已有数据

如果只是新机器全新部署，不需要处理本节。

如果要把旧机器用户和数据迁移过来，不能只迁移代码，还要迁移 Docker volumes。

核心数据：

```text
Postgres 数据卷：
openclaw_pgdata 或 openclaw_pgdata_new

用户 workspace 卷：
hermes-data-<用户短ID>

用户 Hermes home 卷：
hermes-data-<用户短ID>-home
```

用户容器本身可以不迁移。迁移 DB 和 volume 后，gateway 找不到旧容器时会基于 DB 记录和 volume 重建。

旧机器导出示例：

```bash
docker run --rm -v openclaw_pgdata_new:/data -v "$PWD":/backup alpine \
  tar czf /backup/openclaw_pgdata_new.tgz -C /data .

docker volume ls --format '{{.Name}}' | grep '^hermes-data-' > hermes-volumes.txt
while read -r v; do
  docker run --rm -v "$v":/data -v "$PWD":/backup alpine \
    tar czf "/backup/${v}.tgz" -C /data .
done < hermes-volumes.txt
```

新机器恢复示例：

```bash
docker volume create openclaw_pgdata_new
docker run --rm -v openclaw_pgdata_new:/data -v "$PWD":/backup alpine \
  sh -lc 'tar xzf /backup/openclaw_pgdata_new.tgz -C /data'

for f in hermes-data-*.tgz; do
  v="${f%.tgz}"
  docker volume create "$v"
  docker run --rm -v "$v":/data -v "$PWD":/backup alpine \
    sh -lc "tar xzf /backup/$f -C /data"
done
```

恢复后再启动：

```bash
docker compose up -d
```

## 11. 更新部署

远程 main 更新后，在服务器上执行：

```bash
git fetch origin
git checkout main
git pull origin main
```

如果只改了 gateway / frontend：

```bash
python3 deploy_docker.py --rebuild gateway,frontend,manage-front,simple-front --fast
```

如果改了 `hermes-agent/`、`deploy_copy/` 或用户容器运行时依赖：

```bash
bash build_base_image.sh
python3 deploy_docker.py --rebuild hermes,gateway,frontend,manage-front,simple-front --fast
```

重要：重建 `nanobot-hermes-agent:latest` 不会自动替换已有 `hermes-user-*` 容器。已有用户要使用新镜像，需要删除对应用户容器，让 gateway 基于保留的数据卷重建：

```bash
docker rm -f <hermes-user-容器名>
```

不要删除 `hermes-data-*` volume，除非明确要清空用户数据。

## 12. 常见问题

### 用户容器创建失败，提示找不到镜像

检查：

```bash
docker images | grep nanobot-hermes-agent
```

如果没有，重新构建：

```bash
python3 deploy_docker.py --rebuild hermes
```

### LLM 返回 401

优先检查用户容器配置：

```bash
docker exec <hermes-user-容器名> sh -lc 'cat /opt/data/config.yaml'
```

正确方向是：

```yaml
model:
  provider: platform-gateway
  default: openai/glm-5.1
```

如果 provider 不是 `platform-gateway`，检查 `docker-compose.override.yml` 和 gateway env：

```bash
docker exec openclaw-gateway env | grep PLATFORM_DEDICATED_HERMES_DEFAULT_PROVIDER
```

修改后重建或删除旧用户容器，让新配置重新写入。

### 管理后台改默认模型后，新用户默认模型没变

当前管理后台的默认模型配置对用户容器初始化不完全生效。新用户 Hermes 容器默认模型仍主要来自：

```env
DEFAULT_MODEL=openai/glm-5.1
```

已有用户容器也不会因为后台保存模型而自动改 `/opt/data/config.yaml`。

### 只看 `docker logs` 找不到 Hermes 业务错误

Hermes 容器业务日志主要在：

```text
/opt/data/logs/
```

排查用户消息、飞书、LLM 时进入对应 `hermes-user-*` 容器看这个目录。

## 13. 最小部署清单

上线前逐项确认：

- 已拉取私有仓库 `main` 分支。
- `.env` 已配置管理员账号、JWT secret、LLM key、`DEFAULT_MODEL=openai/glm-5.1`。
- `docker-compose.override.yml` 保留 `PLATFORM_DEDICATED_HERMES_DEFAULT_PROVIDER=platform-gateway`。
- 已构建 `hermes-base:latest`。
- 已构建 `nanobot-hermes-agent:latest`。
- `docker compose ps` 中 `postgres`、`gateway`、`frontend`、`manage-front`、`simple-front` 正常运行。
- `curl http://127.0.0.1:9900/api/ping` 返回 `pong`。
- 新注册用户能创建 `hermes-user-*` 容器。
- 新用户容器 `/opt/data/config.yaml` 中 provider 是 `platform-gateway`。
- 安全组没有公网开放 Postgres `9901`。
